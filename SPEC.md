# Team Sheet Data Cruncher — Spec & Handoff

This document is the source of truth for **the cruncher** — a webapp that ingests
Team Sheet Builder staff-sheet PDFs in bulk, extracts the embedded team data, and
produces a spreadsheet and a dashboard. It is written so a fresh session can build
the cruncher from this doc alone.

The **producer** of these PDFs is **Team Sheet Builder**, the sibling repo at
`../teamsheetbuilder`. The embedded formats below are generated by that repo's
`src/pdf/generateTeamSheetPdf.ts` + `src/pdf/teamDataCode.ts`; treat those files
and this doc as the contract.

---

## 1. Goal

- Upload many staff-sheet PDFs at once.
- Extract the machine-readable, **PII-free** team payload embedded on each.
- Expand the compact ids into readable names.
- Emit (a) a **spreadsheet** (CSV / XLSX) and (b) a **dashboard** of usage stats
  (species / item / ability / move / stat-alignment frequencies, etc.).

Only the **Staff Team Sheet** carries data. The Open sheet does not. A combined
"Both" PDF contains the staff page (page 1) plus the open page.

---

## 2. What is embedded in each staff sheet

Every staff sheet carries the same PII-free payload through **two carriers**.
The payload excludes all Player Info by construction — species, ability, item,
moves, Stat Alignment, and the six final stats only.

### 2.1 Digital carrier — transparent text (`TSBv1`) — **use this one**

Real, fully-transparent text drawn on the page. It is invisible on screen and in
print but is returned verbatim by any text extractor (`pdftotext`, PDF.js
`getTextContent`). For **digital PDFs uploaded in bulk, this is the primary and
most reliable extraction path** — no image processing, no external tables needed
to read the raw values (ids are self-describing slugs).

Wire format — one logical payload split across N physical lines so it always
fits the page width:

```
TSBv1~<segmentIndex>~<segmentCount>~<chunk>
```

Reassemble by collecting every `TSBv1~i~n~chunk` match, sorting on `i`, and
concatenating the chunks (tolerant of reordered/interleaved extractor output).
The reassembled payload is:

```
<mon>|<mon>|<mon>|<mon>|<mon>|<mon>          (always six slots, positional)
```

Each `<mon>` is 15 comma-separated fields (empty string = empty field):

```
speciesId, formId, abilityId, itemId, move1, move2, move3, move4,
statAlignmentId, hp, atk, def, spa, spd, spe
```

- The id fields are internal **slug ids** (`incineroar`, `fake-out`,
  `safety-goggles`, `Jolly`, …). Expand them to names via the data files in §4.
- The six stats are the **final displayed stat values** as strings.
- An empty Pokémon slot is all-empty fields (skip it).

### 2.2 Paper carrier — corner QR (`TSBI1`)

A QR in the top-right corner of the staff sheet, for **scanned / photographed**
sheets where the transparent text is lost. Its payload is compact index numbers,
so decoding it requires the code-index registry (§4). Prefer the digital carrier
for uploaded PDFs; treat QR support as a stretch goal (render page → canvas →
QR decode, e.g. `jsQR`).

Wire format (fixed-width, separator-free, all-uppercase base36 so the QR uses
alphanumeric mode):

```
TSBI1 <version:4 chars> <record × 6>
record (29 chars) = species(2) form(2) ability(2) item(2)
                    move1(2) move2(2) move3(2) move4(2)
                    statAlign(1) hp(2) atk(2) def(2) spa(2) spd(2) spe(2)
```

Each 2/1-char field is a base36 **code-index number** (`0` = empty); the six
stats are raw base36 values. Map numbers → slug ids with `code-index.json`
(§4), then slug → name as in §4. The 4-char `version` is the registry's
data-version stamp; warn if it differs from your bundled registry (older data
may not know newer numbers). See the builder's `docs/CODE_INDEX.md` for the full
contract.

## 3. Reference implementation to copy

The builder's `../teamsheetbuilder/scripts/decode_team_data.mjs` is a working Node
decoder for **both** carriers (it uses `pdftotext` for the digital text and the
registry for the QR). Port its parsing logic to the browser (swap `pdftotext` for
PDF.js `getTextContent`, and add canvas+`jsQR` if you implement QR). The exact
sentinels, field order, and widths also live in the builder's
`src/pdf/teamDataCode.ts`.

**PII guardrail:** parse **only** the `TSBv1` / `TSBI1` payload. Do not dump all
page text — the visible sheet still shows the player's name, ID, and DOB. The
embedded payload contains none of that; keep it that way by matching the
sentinel and ignoring everything else.

## 4. Data files needed (bundle a snapshot)

Copy these from the builder's `../teamsheetbuilder/src/data/regulation-mb/` into
the cruncher and version them (they are regulation-specific):

| File | Used for |
| --- | --- |
| `species.json` | slug → `displayName`, `nationalDexNumber`, `types`, base/presented stats |
| `moves.json` | slug → `displayName`, `type`, `category` |
| `abilities.json` | slug → `displayName` |
| `items.json` | slug → `displayName` |
| `stat-alignments.json` | slug → `displayName`, `raises`/`lowers` |
| `code-index.json` | number ↔ slug (only if decoding the QR carrier) |

Each record has an `id` (the slug) and a `displayName`. Build `id → record`
maps for O(1) expansion. For the QR, build `number → slug` per category from
`code-index.json` (see the builder's `docs/CODE_INDEX.md`).

Keep these in sync when the regulation data is re-exported. The QR's version
stamp is your signal that a sheet was produced with a newer registry than the
cruncher has bundled.

## 5. Output

### 5.1 Spreadsheet (CSV + XLSX)

Recommended: **one row per Pokémon** (six per team), plus a source column so
teams can be reconstructed. Suggested columns:

```
source_file, slot (1-6), species, form, ability, item,
move1, move2, move3, move4, stat_alignment, hp, atk, def, spa, spd, spe
```

Optionally a second sheet with **one row per team** (species 1-6 across
columns) for quick scanning. Export via CSV and/or SheetJS (`xlsx`).

### 5.2 Dashboard

Aggregate across all uploaded sheets. Core metrics:

- **Species usage** — count and % of teams running each species (this is the
  headline VGC metric). Sort desc, show top N.
- **Item / ability / move / stat-alignment usage** — same treatment.
- **Move usage per species** (drill-down) and **item-on-species** pairings.
- Totals: sheets processed, teams, unique species, parse failures.

Charts: horizontal bar charts for usage rankings; a table view with sortable
counts. Follow the design guidance in the `dataviz` skill if available.

## 6. Recommended stack & constraints

Mirror Team Sheet Builder so the two feel like a suite:

- **Vite + React + TypeScript**, static/client-side, GitHub Pages-hostable.
- **PDF.js (`pdfjs-dist`)** for in-browser text extraction (`getTextContent`).
- **`xlsx` (SheetJS)** for spreadsheet export; plain CSV as a fallback.
- A charting lib for the dashboard (Recharts or similar).
- Optional QR: `jsqr` + canvas render of the page.

**Process everything in the browser.** The uploaded PDFs contain visible PII
(player names/IDs/DOB), so keeping extraction client-side means the files never
leave the device — matching Team Sheet Builder's no-backend, privacy-first
stance. No accounts, no server, no upload endpoint.

---

## 7. Edge cases to handle

- **No `TSBv1`/`TSBI1` payload found** — the PDF isn't a Team Sheet Builder staff
  sheet (or is an Open sheet). Report it as skipped, don't crash the batch.
- **Empty Pokémon slots** — species field empty; skip that slot.
- **Unknown slug/number** — data files are older than the sheet. Show the raw
  id and flag it; for the QR, compare the version stamp and warn.
- **Duplicate/blank stats** — stats are strings; keep as-is. Empty stats are
  possible on partially-filled sheets — don't assume all six are present.
- **Large batches** — parse incrementally with progress; PDF.js is async.
- **Mixed uploads** — some PDFs may be scans (QR only) and some digital (text).
  Try text first, fall back to QR if implemented.

---

## 8. Acceptance criteria

- Drag-and-drop / select many PDFs; a progress indicator during parsing.
- Each valid staff sheet yields six (or fewer) decoded Pokémon with names.
- No Player Info ever appears in the output.
- Downloadable CSV and XLSX of the aggregated data.
- A dashboard with at least species/item/ability/move usage rankings.
- Runs fully client-side and can deploy to static hosting.
